1. Introducción

1.1 Motivación

La actividad física regular es reconocida como uno de los pilares fundamentales de la salud preventiva. Sin embargo, la planificación de rutinas de entrenamiento personalizadas suele requerir conocimientos especializados o herramientas costosas. La posibilidad de contar con un sistema que automatice la generación de rutinas a partir de un catálogo de ejercicios configurable representa un aporte concreto tanto para usuarios independientes como para instructores que administran múltiples planes de entrenamiento.

Desde el punto de vista formativo, el desarrollo de este sistema ofrece la oportunidad de integrar conceptos fundamentales de programación orientada a objetos —encapsulamiento, herencia, interfaces y polimorfismo— con patrones de diseño consolidados y buenas prácticas de arquitectura de software.

1.2 Objetivo del trabajo

El objetivo principal es diseñar e implementar una aplicación de escritorio con interfaz gráfica (Java Swing) que permita gestionar un catálogo de ejercicios cargado desde archivo CSV o base de datos SQLite, generar rutinas de entrenamiento personalizadas respetando una restricción de no repetición semanal, y revisar la rutina generada ejercicio por ejercicio hasta obtener un resumen final.

Los objetivos específicos son:

  • Aplicar el patrón Observer para desacoplar la lógica de negocio de la interfaz gráfica.
  • Modularizar la solución en los paquetes backend y frontend.
  • Implementar validación robusta de datos mediante una excepción personalizada.
  • Persistir el historial semanal en disco para garantizar variedad en semanas consecutivas.

1.3 Estructura del documento

El presente informe se organiza de la siguiente manera: la Sección 2 describe la solución propuesta, detallando las clases diseñadas, las relaciones entre ellas y las decisiones de diseño que las justifican, incluyendo el diagrama de clases. La Sección 3 presenta las conclusiones del trabajo, evaluando el cumplimiento de los objetivos y los aprendizajes obtenidos.

2. Descripción de la Solución

2.1 Modelo propuesto

La solución se articula en torno a una arquitectura en dos capas claramente diferenciadas: el paquete backend, que contiene toda la lógica de negocio (modelos, carga, generación e historial), y el paquete frontend, responsable exclusivo de la interfaz gráfica Swing. La comunicación entre ambas capas se realiza de forma asincrónica mediante el patrón de diseño Observer (notificación-suscripción), garantizando que el backend nunca dependa ni referencie directamente las ventanas de la interfaz.

El flujo principal de la aplicación es lineal: el usuario carga un catálogo de ejercicios, configura los requisitos de la rutina, navega por los ejercicios generados y obtiene un resumen final. Cada etapa corresponde a una ventana Swing independiente, lo que favorece la cohesión y facilita el mantenimiento.

2.4 Justificación de decisiones de diseño

Decisión Alternativa descartada Justificación
Patrón Observer para comunicación entre capas Llamadas directas del backend a métodos de las ventanas El Observer garantiza el principio de inversión de dependencias: el backend no conoce ni depende de las clases Swing. Permite agregar o eliminar ventanas sin modificar el Controlador.
Notificación en el EDT mediante SwingUtilities.invokeLater Actualizar la UI directamente desde el hilo de carga Swing no es thread-safe. Todas las modificaciones a componentes gráficos deben realizarse en el Event Dispatch Thread; de lo contrario se producen condiciones de carrera y comportamientos indefinidos.
Singleton en el Controlador Pasar el controlador como parámetro a cada ventana Al existir un único controlador compartido, todas las ventanas acceden al mismo estado de la aplicación sin necesidad de sincronización adicional ni de pasar referencias entre constructores.
Enums para TipoEjercicio y NivelIntensidad Cadenas de texto libres o constantes enteras Los enums restringen el dominio de valores en tiempo de compilación, eliminando errores por valores inválidos y habilitando el uso de switch exhaustivo y colecciones como EnumMap para mayor eficiencia.
ExcepcionCarga verificada con enum de tipos de error Excepción genérica o código de retorno entero La excepción verificada obliga al compilador a que el código cliente la maneje. El enum interno Tipo permite reaccionar distinto ante cada categoría de error (archivo no encontrado, formato inválido, etc.) sin parsear el mensaje de texto.
Lista inmutable en Rutina Lista mutable accesible directamente Encapsular los ejercicios como lista inmutable (Collections.unmodifiableList) garantiza que ningún componente externo pueda modificar la rutina generada, respetando el principio de encapsulamiento.
Persistencia del historial en archivo de texto plano Base de datos separada o solo en memoria El archivo de texto es suficiente para la restricción semanal y no requiere dependencias adicionales. Persiste entre ejecuciones del programa, lo que permite cumplir la restricción aunque el usuario reinicie la aplicación.
Detección automática del delimitador en el CSV Exigir siempre el mismo delimitador Los archivos CSV generados por diferentes programas (Excel, LibreOffice, etc.) pueden usar coma o punto y coma. La detección automática mejora la usabilidad sin añadir complejidad al usuario.
CopyOnWriteArrayList para la lista de observadores ArrayList con bloque synchronized Los hilos de carga notifican mientras el EDT puede suscribir o desuscribir ventanas. CopyOnWriteArrayList es thread-safe para estas operaciones sin requerir bloqueos explícitos, reduciendo el riesgo de deadlocks.

2.5 Descripción del flujo de la aplicación

El sistema sigue un flujo de cinco etapas correspondientes a sus cinco ventanas:

  1. Carga de ejercicios (VentanaCarga): el usuario selecciona la fuente de datos (CSV o SQLite) y la ruta del archivo. El Controlador lanza un hilo de fondo que delega en CargadorEjercicios. Al finalizar, notifica EJERCICIOS_CARGADOS o ERROR_CARGA a todos los observadores suscritos.
  2. Visualización de estadísticas (VentanaPrincipal): al recibir EJERCICIOS_CARGADOS, la ventana principal actualiza las tarjetas con el total de ejercicios, el tiempo disponible, la distribución por tipo y por nivel de intensidad, y habilita el botón de generación.
  3. Generación de rutina (VentanaGeneracion): el usuario configura cuántos ejercicios de cada tipo e intensidad desea. El Controlador carga el historial semanal, invoca a GeneradorRutina y persiste el nuevo historial. Notifica RUTINA_GENERADA con la Rutina resultante.
  4. Revisión ejercicio a ejercicio (VentanaRevision): el usuario navega por la rutina con los botones "Volver" y "Siguiente". En el último ejercicio, el botón cambia a "Resumen de la rutina".
  5. Resumen final (VentanaResumen): muestra la distribución estadística de la rutina generada: totales por tipo, por intensidad y tiempo estimado total.

3. Conclusión

3.1 Reflexión sobre el trabajo realizado

El desarrollo del sistema de rutinas de entrenamiento permitió poner en práctica de forma integrada los principios de la programación orientada a objetos, la programación orientada a eventos y los patrones de diseño. La división en paquetes backend y frontend resultó ser una guía de organización efectiva que mantuvo bajo control la complejidad del proyecto, facilitando la localización de errores y la extensión del sistema.

La implementación del patrón Observer representó el mayor desafío técnico, pues requirió comprender cómo coordinar hilos de fondo con el Event Dispatch Thread de Swing sin incurrir en condiciones de carrera. El uso de SwingUtilities.invokeLater demostró ser la solución estándar y robusta para este problema, y su comprensión profunda constituye un aprendizaje transferible a cualquier aplicación Java con interfaz gráfica.

3.2 Cumplimiento de objetivos

El sistema cumple con todos los requerimientos establecidos: carga desde archivo CSV y base de datos SQLite con validación completa, generación de rutinas configurables por tipo e intensidad, restricción de no repetición semanal persistida en disco, navegación con estado entre ejercicios, y cinco ventanas Swing diferenciadas. La comunicación entre capas se realiza exclusivamente a través del mecanismo de notificación-suscripción, sin referencias directas del backend al frontend.

3.3 Aprendizajes obtenidos

  • Patrón Observer aplicado: se comprendió cómo implementar el desacoplamiento entre productor y consumidor de eventos, y la importancia de gestionar correctamente el ciclo de vida de los observadores (suscripción y desuscripción) para evitar fugas de memoria.
  • Multithreading en aplicaciones de escritorio: se aprendió que las operaciones costosas deben ejecutarse en hilos separados para no bloquear la interfaz, y que la actualización de componentes Swing debe ser siempre redirigida al EDT.
  • Diseño de excepciones personalizadas: el uso de una excepción con un enum interno de tipos resultó más expresivo y mantenible que el manejo genérico de errores, permitiendo al código cliente tomar decisiones específicas según la causa del fallo.
  • Separación de responsabilidades: mantener las clases con una única responsabilidad bien definida (un cargador que solo carga, un generador que solo genera, un controlador que coordina) redujo la complejidad de cada clase individualmente y facilitó las pruebas y modificaciones.
  • Persistencia liviana: la restricción semanal pudo implementarse de forma simple con un archivo de texto plano, demostrando que no siempre es necesario recurrir a una base de datos completa para resolver problemas de persistencia sencillos.
